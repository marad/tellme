## Shared design

NEW CODE on top of FEAT-0001's daemon and FEAT-0002's streaming. Three
small-but-related additions:

1. The CLI gains the ability to auto-spawn the daemon when no daemon
   is reachable at the expected socket path (AC-1).
2. The daemon gains a per-instance idle-exit timer that fires when the
   queue has been continuously empty for a configured interval (AC-2).
3. Both behaviors are gated by an opt-out env var (`TELLME_NO_DAEMON`)
   that bypasses the daemon entirely (AC-3).

The wire protocol, queue, worker, and streaming machinery from
FEAT-0001 / FEAT-0002 are unchanged.

### CLI auto-spawn helper

`src/cli/daemon-cmd.ts` (modified). Today's `daemonStart` is the
user-facing `tellme daemon start` command — it prints "daemon started
(socket: …)". We extract the spawn-and-wait logic into a small
programmatic helper that returns success/failure silently:

```
async function ensureDaemonRunning(opts?: {
    spawner?: () => void;          // defaults to production detached spawn
    waitMs?: number;               // default 5000
}): Promise<boolean>
```

Behavior:

1. Probe via `statusPing` — if a daemon already answers, return `true`
   immediately.
2. Call `spawner` (default: today's `spawn(process.execPath, [bin,
   "__daemon-main__"], { detached: true, stdio: "ignore" })` +
   `proc.unref()`).
3. Poll `statusPing` every 100ms for up to `waitMs`. Return `true` on
   first success; return `false` on timeout.
4. Does not print anything. Logging is the caller's job.

`daemonStart` is rewritten to call `ensureDaemonRunning`, then print
the appropriate user-facing message based on the boolean.

### CLI routing predicate

`src/cli/daemon-routing.ts` (NEW, ~10 lines). A single pure function:

```
export function shouldUseDaemon(env: NodeJS.ProcessEnv = process.env): boolean {
    return env.TELLME_NO_DAEMON !== "1";
}
```

Trivial enough to inline, but extracting it gives AC-3 a unit-testable
gate and lets future config keys plug in without touching CLI flow.

### CLI entry-point wiring

`src/cli/index.ts` (modified). The streaming and one-shot daemon
branches both gain the same shape:

```
if (shouldUseDaemon()) {
    if (!existsSync(getSocketPath())) {
        await ensureDaemonRunning();   // best-effort; if it fails we fall through
    }
    const code = await tryDaemonStreaming(...);  // or tryDaemonRoute(...)
    if (code !== null) process.exit(code);
}
// fall through to in-process path
```

If `shouldUseDaemon()` is `false`, both branches are skipped — neither
auto-spawn nor `tryDaemon*` runs, and the CLI proceeds straight to the
in-process synthesis that exists today (AC-3).

If auto-spawn fails (timeout, early exit), the call falls through to
in-process — preserving today's "fall back gracefully" shape.

### Idle exit

`src/core/daemon-server.ts` (modified). One new env-var-driven timer
inside `startDaemon`:

```
const daemonIdleMs = (() => {
    const raw = process.env.TELLME_DAEMON_IDLE_MS;
    const n = raw ? parseInt(raw, 10) : NaN;
    return Number.isFinite(n) && n > 0 ? n : 600000;   // 10 min default
})();
```

Lifecycle:

- `armIdleTimer()` — start a `setTimeout(stop, daemonIdleMs)` if and
  only if `queue.length === 0`, `activeItem === null`, and
  `!shuttingDown`. Idempotent (if the timer is already armed, no-op).
- `disarmIdleTimer()` — clear it.
- Wire-up:
  - Call `armIdleTimer()` once after `server.listen` callback (so a
    fully idle daemon will exit even with no traffic).
  - In `kickWorker`: `disarmIdleTimer()` (about to do work).
  - In the worker's outer `while` loop, when the queue empties and the
    worker is about to exit (`workerRunning = false`): `armIdleTimer()`.
  - In `handleConnection`'s entry: `disarmIdleTimer()` (a connection
    is mid-flight; even a `status` probe should keep us alive long
    enough to answer it). Re-arm in the connection's `finally` once
    the socket has been closed AND the queue is empty AND no active
    item — guard via the same `armIdleTimer()` predicates.
- `stop()` already does the right thing (closes server, unlinks
  socket and pid file, frees engine, resolves exit). The idle path
  reuses it as-is, so a subsequent CLI invocation will find no socket
  and trigger auto-spawn (AC-1).

Default 10 min is conservative — tuned more for "don't sit on 430MB
forever on idle laptops" than aggressive recycling. Tests override to
~50ms.

## AC-1 (added): auto-spawn on first invocation when daemon is absent

Approach: Per the shared design — `src/cli/index.ts` checks
`shouldUseDaemon()` and, if there is no socket file at
`getSocketPath()`, calls `ensureDaemonRunning()` before invoking
`tryDaemonStreaming` / `tryDaemonRoute`. `ensureDaemonRunning` reuses
`daemonStart`'s existing detached-spawn pattern (`spawn(process.execPath,
[bin, "__daemon-main__"], { detached: true, stdio: "ignore" })` +
`proc.unref()`) and polls `statusPing` for readiness. On success, the
CLI proceeds with the existing daemon route (zero behavior change from
the daemon's perspective). On timeout/early-exit, the CLI falls
through to in-process synthesis. Subsequent invocations find the
socket already present; auto-spawn is a no-op (the readiness probe
short-circuits).

Test: `it("AC-1: ensureDaemonRunning spawns and waits for readiness ...")`
in `src/cli/__tests__/daemon-cmd.test.ts`. Override
`TELLME_DAEMON_DIR` to a tmp dir. Inject a `spawner` that calls
`void startDaemon({ ttsFactory: fakeFactory, installSignalHandlers:
false })` — runs the daemon in-process with the fake engine. Assert
`ensureDaemonRunning()` returns `true` within the wait window. Then
call again with no spawner override and assert it returns `true`
immediately (already-running short-circuit).

Files: `src/cli/daemon-cmd.ts`, `src/cli/index.ts`,
`src/cli/__tests__/daemon-cmd.test.ts`.

## AC-2 (added): daemon exits after idle interval

Approach: Per the shared design — `startDaemon` reads
`TELLME_DAEMON_IDLE_MS` (default 600000ms). An `armIdleTimer` /
`disarmIdleTimer` pair guards a single `setTimeout(stop,
daemonIdleMs)`. Armed only when the queue is empty, no item is
active, no connection is mid-flight, and the daemon isn't already
shutting down. Disarmed on `kickWorker`, on every incoming connection,
and on `stop`. Re-armed when the worker drains the queue and when a
connection's reader loop exits leaving the daemon idle. On expiry
the timer calls the existing `stop()` — which closes the server,
unlinks the socket, unlinks the pid file, frees the engine, and
resolves the exit promise. The next CLI invocation sees no socket and
auto-spawns a fresh daemon (AC-1).

Test: `it("AC-2: daemon exits cleanly after idle interval ...")` in
`src/core/__tests__/daemon-server.test.ts`. Override
`TELLME_DAEMON_IDLE_MS=50`. Start daemon. Do nothing. Wait ~150ms.
Assert: socket file at `getSocketPath()` no longer exists, pid file
no longer exists, and the handle's exit promise has resolved (the
daemon's `await runDaemon()` returns). Second test: `it("AC-2:
activity resets the idle timer ...")` — start daemon, send a
`status` ping every 30ms for 200ms (well past 50ms idle), assert
socket still exists; then stop pinging, wait ~150ms, assert socket
gone.

Files: `src/core/daemon-server.ts`,
`src/core/__tests__/daemon-server.test.ts`.

## AC-3 (added): TELLME_NO_DAEMON=1 bypasses daemon entirely

Approach: Per the shared design — `shouldUseDaemon(env)` returns
`false` iff `env.TELLME_NO_DAEMON === "1"`. `src/cli/index.ts` gates
both daemon branches (streaming and one-shot) on this predicate. When
opt-out is active, neither `ensureDaemonRunning` nor `tryDaemon*` runs;
the CLI proceeds directly to today's in-process synthesis path
(`prepareForSpeech`, `TellMeTts`, `createStreamingPlayer`/`playAudio`),
matching pre-FEAT-0001 behavior.

Test: `it("AC-3: shouldUseDaemon respects TELLME_NO_DAEMON=1 ...")`
in `src/cli/__tests__/daemon-routing.test.ts`. Direct unit test of
the predicate against a synthetic env object: `{}` → true,
`{TELLME_NO_DAEMON: "1"}` → false, `{TELLME_NO_DAEMON: "0"}` → true,
`{TELLME_NO_DAEMON: "true"}` → true (only the literal string `"1"`
opts out — same convention as `TELLME_TEST_SILENT`). A second
assertion in the integration test from AC-1 confirms that with
`TELLME_NO_DAEMON=1` set, `ensureDaemonRunning` is *not* called by
the gating code (verified by source-grep / by wrapping the helper in a
spy passed via DI — easiest is the predicate-only unit test plus a
code review).

Files: `src/cli/daemon-routing.ts`, `src/cli/index.ts`,
`src/cli/__tests__/daemon-routing.test.ts`.
