# Sync plan — FEAT-0002 TTS daemon streaming protocol

Started: 2026-05-02T17:17:33.449Z
Snapshot state: new
Drift summary: 5 added (whole spec)

## Shared design

NEW CODE on top of FEAT-0001's daemon. The wire envelope, framing, queue,
worker, and stop semantics are reused unchanged. FEAT-0002 extends the
client→server message set so a single connection can deliver text
incrementally, and changes the daemon's per-request handler to start
playback as soon as a sentence boundary is detected.

### Protocol additions

`src/core/daemon-protocol.ts` (modified):

- `SpeakRequest` gains an optional `streaming?: boolean`. Default/absent = one-shot
  (today's behavior). When `true`, the daemon keeps reading from the socket
  for follow-up messages instead of returning after queueing.
- New `ChunkMessage`: `{kind: "chunk", text: string}` — appends text to the
  buffer of the in-flight streaming request on this connection. Sent with
  no `version` (the version handshake already happened on the speak frame).
- New `EndMessage`: `{kind: "end"}` — graceful end-of-input. Tells the daemon
  to play out any remaining buffered text and signal `done`.
- Both `chunk` and `end` are added to `ClientMessage`.

A streaming request's first frame is still a `speak`. Its `text` field, if
non-empty, is treated as the first chunk (saves a roundtrip for callers
that already have the first piece). Subsequent `chunk` frames append.

Server→client messages are unchanged. `done` / `error` / `stopped` already
cover all completion paths a streaming connection needs.

### Sentence-boundary buffering

NEW: `src/core/sentence-buffer.ts`. A small stateful buffer with two methods:

- `push(text)`: append; return `string[]` of every complete sentence
  newly available (each one already trimmed). A "complete sentence" is
  text up to and including a `.`, `!`, or `?` followed by whitespace or
  end-of-buffer when the next non-whitespace char is uppercase / a digit
  / EOF. Implementation is a forward scan — keeps buffer state across
  pushes so a chunk that ends mid-sentence (`"Hello wor"`, then `"ld."`)
  emits once on the second push.
- `flush()`: return whatever residue remains as a final sentence (used
  on `end`, idle timeout, or hard duration cap).

Each emitted sentence is then run through `prepareForSpeech` (unless
`raw`) and synthesized via `engine.speak`. Re-running text-prep per
sentence is acceptable for FEAT-0002 — multi-sentence markdown
constructs (e.g. fenced code blocks) naturally suppress sentence-end
detection inside them, so they buffer until the close fence arrives.

Unit tests live in `src/core/__tests__/sentence-buffer.test.ts`.

### Streaming queue item

`src/core/daemon-server.ts` (modified). A `QueueItem` is extended:

```
interface QueueItem {
    req: SpeakRequest;
    socket: Socket;
    resolved: boolean;
    streaming?: {
        // bounded async channel from the connection reader to the worker
        push(text: string): void;
        end(): void;        // graceful end-of-input
        kill(): void;       // hard close — drop any not-yet-played buffer
        // worker awaits this iterator
        [Symbol.asyncIterator](): AsyncIterator<string | { done: true }>;
    };
}
```

The connection handler, on a `speak` with `streaming: true`:

1. Builds the streaming channel.
2. Pushes the speak frame's `text` (if non-empty) as the first chunk.
3. Enqueues the item, sends `ack`, kicks the worker.
4. Continues reading from the socket: `chunk` → `streaming.push`,
   `end` → `streaming.end`, anything else → close.
5. If the socket closes from the client side before `end`, treat that
   as `streaming.end` (graceful — covers the "client disconnect"
   completion path called out in AC-5).

The worker's `processItem` branches on `streaming`:

- One-shot path: unchanged from FEAT-0001.
- Streaming path: drain the channel through `SentenceBuffer`. As each
  sentence emerges, call `engine.speak` for it. After the channel is
  closed, flush the buffer and synthesize the residue. Then signal
  completion (`done` / `stopped`) the same as one-shot.

Sentences within one streaming connection play back-to-back through the
same `engine.speak` machinery, so the single-output-stream invariant
from FEAT-0001 is preserved.

### Timers

Two configurable per-connection timers (env vars, read at daemon
startup, with sane defaults):

- `TELLME_STREAM_IDLE_MS` — idle timeout (default 30000). Reset on every
  `chunk`. On expiry: call `streaming.end` (play out everything
  buffered, then signal `done`). The daemon does NOT abort mid-utterance.
- `TELLME_STREAM_MAX_MS` — hard duration cap from first chunk
  (default 300000). On expiry: call `streaming.end` so the worker plays
  out what's already been buffered. Critically, the connection handler
  also stops accepting more `chunk` messages from the socket (further
  chunks are silently dropped) — this is the "stops accepting more
  chunks, finishes the sentence in flight, closes" behavior in AC-3.

Both timers are cleared when the worker finishes the request.

For tests: env-var driven configuration lets tests use small values
(e.g. 50ms idle, 200ms max).

### Stop while streaming (AC-4)

`handleStop` in `daemon-server.ts` is extended: in addition to clearing
the queue and aborting the active sink, if the active item is a
streaming item, call `streaming.kill()` so the worker stops feeding
sentences to the engine, and `socket.destroy()` so the writer can no
longer send chunks. The `stopped` reply is written before destroy.

`tryDaemonRoute` in `daemon-client.ts` (modified) and the CLI's stdin
forwarder (modified `src/cli/index.ts`) handle EPIPE / ECONNRESET on
chunk writes by treating them as "daemon closed us" → exit 130 (the
same code the existing one-shot path already returns for `stopped`).
No stack trace.

### CLI streaming forwarder

`src/cli/index.ts` (modified):

- When stdin is a TTY or a positional arg was given, behavior is
  unchanged (one-shot through the daemon, or in-process fallback).
- When stdin is a pipe (`!process.stdin.isTTY` and no positional arg):
  open the daemon connection, send a `speak` with `streaming: true` and
  empty text, then forward stdin chunks as `chunk` frames as they arrive,
  send `end` on stdin EOF.

`tryDaemonRoute` gains a streaming entry point or a flag — concretely,
the CLI passes either `text: string` (one-shot) or `stdin: Readable`
(streaming) and the client picks the right path. Either way it returns
the same exit-code shape as before.

The in-process fallback path is unchanged (out of scope for this spec —
it currently buffers stdin via `readStdin()` and that stays).

## AC-1 (added): streaming connection plays sentence-by-sentence

Approach: Per the shared design — `SpeakRequest.streaming: true` opens a
streaming connection. The connection handler keeps reading frames; the
worker processes the request through a channel + `SentenceBuffer`,
calling `engine.speak` per emitted sentence. Playback begins on the
first detected sentence boundary, before `end` arrives. All sentences
from one connection feed the same `engine.speak` pipeline back-to-back
through the same audio sink, so they play contiguously.

Test: `it("AC-1: streaming connection starts playback before end-of-input ...")`.
Open a streaming connection, push two sentences as separate chunks
(`"Hello world. "` then `"Second sentence."`), wait briefly, assert
the fake engine has been called for `"Hello world."` *before* `end` is
sent. Then send `end`, assert second sentence is synthesized and `done`
is received.

Files: `src/core/daemon-protocol.ts`, `src/core/sentence-buffer.ts`,
`src/core/daemon-server.ts`, `src/core/daemon-client.ts`,
`src/cli/index.ts`, `src/core/__tests__/sentence-buffer.test.ts`,
`src/core/__tests__/daemon-server.test.ts`.

## AC-2 (added): idle timeout plays out, does not discard

Approach: Per-connection idle timer (`TELLME_STREAM_IDLE_MS`, default
30000ms; reset on each chunk arrival). On expiry the connection handler
calls `streaming.end()` — exactly as if the client had sent `end`.
The worker plays out whatever is buffered (including a sentence in
flight and any tail from `flush()`), signals `done`, closes. The
daemon does NOT call `streaming.kill()` on idle, so no buffered text
is dropped.

Test: `it("AC-2: idle timeout drains buffer and signals done ...")`.
Set `TELLME_STREAM_IDLE_MS=50` for the daemon. Push `"Hello world. "`
and one chunk that ends mid-sentence (`"And second"`), then stop sending
without `end`. After ~80ms assert: the engine was called for both
`"Hello world."` and the residue `"And second"` (flush emits the tail),
and the client received `done`.

Files: `src/core/daemon-server.ts`, `src/core/__tests__/daemon-server.test.ts`.

## AC-3 (added): hard duration cap finishes current sentence and closes

Approach: Per-connection max-duration timer (`TELLME_STREAM_MAX_MS`,
default 300000ms; starts on the speak frame). On expiry: (1) the
connection handler stops processing further `chunk` frames (subsequent
chunks are silently discarded — the connection no longer feeds the
buffer); (2) the handler calls `streaming.end()` so the worker drains
what's already been buffered (which includes anything currently in
the engine and any residue in the SentenceBuffer); (3) when the worker
returns, the connection is closed normally (`done`).

Test: `it("AC-3: hard duration cap stops accepting chunks and closes ...")`.
Set `TELLME_STREAM_MAX_MS=100` for the daemon and use a fake engine
with `playMs: 50`. Push `"First sentence. "` immediately, then push
another chunk after 150ms (after the cap fires). Assert the engine was
called for `"First sentence."` but NOT for the late chunk; client
received `done` (not `error`).

Files: `src/core/daemon-server.ts`, `src/core/__tests__/daemon-server.test.ts`.

## AC-4 (added): stop closes streaming connection cleanly

Approach: `handleStop` on a streaming active item calls
`streaming.kill()` (worker stops feeding sentences and exits the
streaming branch), writes `{kind: "stopped"}`, then `socket.destroy()`.
The client side: in `tryDaemonRoute`'s streaming path, the chunk
writer handles `EPIPE` / `ECONNRESET` errors (and the writeMessage
promise rejection they produce) by treating the connection as closed
and exiting with code 130. No stack trace bubbles to the user.

Test: `it("AC-4: stop while streaming closes the connection cleanly ...")`.
Open a streaming connection, push enough text to start playback,
then from a second connection send `{kind: "stop"}`. Assert the
streaming connection's reader yields a `stopped` message before EOF,
and that any subsequent attempt to write a `chunk` to that socket
either rejects or is silently absorbed without throwing
unhandled. Also assert the next speak from a fresh connection works
(the daemon is not wedged).

Files: `src/core/daemon-server.ts`, `src/core/daemon-client.ts`,
`src/cli/index.ts`, `src/core/__tests__/daemon-server.test.ts`.

## AC-5 (added): queued requests wait for streaming connection to complete

Approach: No new code — falls out of the existing single-worker
invariant from FEAT-0001 (`worker()` processes items one at a time;
the streaming branch of `processItem` returns only when the
connection has signaled `end` / `kill` / idle / max). New requests
push to `queue` and wait. Cover with a test for each completion path.

Test: `it("AC-5: a second request waits for the streaming connection ...")`.
Open a streaming connection, push two sentences, do NOT send `end`.
Open a second one-shot speak. Assert the second one's engine call
starts strictly after the streaming connection's last engine call
finishes (regardless of which completion path ends the streamer —
parameterize the test over: graceful `end`, idle timeout (set
`TELLME_STREAM_IDLE_MS=50`), client disconnect (`socket.destroy()`
from client side)).

Files: `src/core/__tests__/daemon-server.test.ts`. (No production
changes — this AC is exercised by the queue from FEAT-0001 already.)

## AC-7 (added): single language per streaming connection

Approach: Today `synthesize` (`src/core/daemon-server.ts:418-457`)
re-runs `detectLanguage(text)` per sentence whenever
`merged.language === "auto"`. That violates AC-7's requirement that
auto-mode pick one language for the whole connection.

Change: hoist the language decision out of `synthesize` into the
streaming-branch scope. Cache `connectionLanguage: "en" | "pl" | null`
alongside `currentSink` / `currentRate`. On the first sentence:

- if `merged.language` is `"en"` or `"pl"`, set
  `connectionLanguage = merged.language`;
- if `merged.language === "auto"`, set
  `connectionLanguage = detectLanguage(firstSentence)`.

On subsequent sentences, reuse `connectionLanguage` directly — no
detection, no re-evaluation. The cache is per-connection (lives in
the same scope as `currentSink`) so a fresh streaming connection
re-detects.

Tests:

1. `AC-7: auto-mode detects language once on the first sentence and
   reuses it`. Feed an English first sentence followed by a
   Polish-leaning second sentence with `lang: "auto"`. Assert both
   `engine.speak` calls received `language: "en"` (the first
   sentence's detection) by inspecting `engine.calls[i].language`.
2. `AC-7: explicit lang override applies to every sentence`.
   Open a streaming connection with `lang: "pl"`, push two sentences
   whose content would auto-detect differently. Assert both
   engine.speak calls received `language: "pl"`.

Files: `src/core/daemon-server.ts`,
`src/core/__tests__/daemon-server.test.ts`.
