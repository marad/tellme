## Shared design

NEW CODE. The daemon and CLI share a small wire protocol over a Unix domain socket at `~/.tellme/daemon.sock`. Messages are length-prefixed JSON: a 4-byte big-endian length followed by a UTF-8 JSON object. The protocol is intentionally minimal — only what FEAT-0001 needs. FEAT-0002 will extend the same envelope with streaming chunk messages.

Client → server messages:

- `{kind: "speak", version: 1, text: string, lang?: "auto"|"en"|"pl", voice?: string, speed?: number, raw?: boolean}` — speak the text and report when playback finishes.
- `{kind: "stop", version: 1}` — halt current utterance and clear the queue.
- `{kind: "status", version: 1}` — report daemon status.

Server → client messages:

- `{kind: "ack"}` — request accepted (sent immediately after queueing).
- `{kind: "done", ok: true}` — playback finished successfully.
- `{kind: "error", message: string}` — synthesis or playback failed.
- `{kind: "stopped"}` — stop request handled.
- `{kind: "status", running: true, socketPath: string, queueDepth: number, version: 1}` — status reply.
- `{kind: "version-mismatch", expected: 1, got: number}` — protocol version refusal.

The `version` field is the protocol version (currently `1`). Either side that sees a mismatch sends `version-mismatch` and closes — covers AC-3.

The daemon is a single Node process. On startup it loads the TTS models once (fulfilling AC-5) and opens the socket. It maintains a FIFO queue of pending speak-requests and processes them one at a time on a single worker. While the worker plays an utterance, additional speak-requests queue (fulfilling AC-4). A `stop` message clears the queue and aborts the active player handle (fulfilling AC-6). The daemon stays running until it receives an explicit `daemon stop` from the CLI or its process is killed (no auto-shutdown — that's FEAT-0003).

The CLI gains a thin client layer that:

1. Detects whether `~/.tellme/daemon.sock` is alive (connect-and-handshake; remove stale sockets on connect-refused).
2. If alive, sends a `speak` request with the user's args and waits for `ack` then `done` or `error`.
3. If not alive, falls back to today's in-process flow unchanged.

The CLI grows a `daemon` subcommand with `start | stop | status` (AC-11). `start` forks a detached child running the daemon. `stop` connects, sends a stop+shutdown (or just kills the PID), waits for socket to disappear. `status` connects, sends `{kind: "status"}`, prints the reply.

## AC-1 (added): route to daemon when running

Approach: In `src/cli/index.ts`, before constructing `TellMeTts`, call `tryDaemonRoute(args, config)` from the new `daemon-client.ts`. If the socket exists and a handshake succeeds, the function sends a `speak` request, awaits completion, and exits with the appropriate code. If the socket is absent or connect refused, it returns `null` and the existing in-process path runs as before. Crucially, `tryDaemonRoute` returns *before* any TTS engine is constructed, so the CLI process never loads models when the daemon is doing the work.

Files: `src/core/daemon-client.ts` (new), `src/cli/index.ts` (modified).

## AC-2 (added): per-user UDS, no network

Approach: `daemon-paths.ts` resolves the socket to `${configDir}/daemon.sock` where `configDir` is the existing `~/.tellme/` (already used by config). The daemon binds via `net.createServer().listen(socketPath)` — Node's `net` over a path is UDS by construction; no `host:port` is ever used. Permissions are set to 0600 after bind. Tests assert the path resolves under the user's home and that no `host` argument is passed.

Files: `src/core/daemon-paths.ts` (new), `src/core/daemon-server.ts` (new).

## AC-3 (added): incompatible version → clear error

Approach: Both client and server check `msg.version` on the first message they receive. Mismatch → send `{kind: "version-mismatch", expected: PROTOCOL_VERSION, got}` and close. The CLI surfaces this as `Error: tellme daemon protocol mismatch (CLI v1, daemon vN). Restart the daemon: tellme daemon stop && tellme daemon start.` Returns nonzero exit.

Files: `src/core/daemon-protocol.ts` (new — defines `PROTOCOL_VERSION`), `src/core/daemon-server.ts` (new), `src/core/daemon-client.ts` (new).

## AC-4 (added): queue, no overlap

Approach: Daemon holds a `Queue<Request>`. The playback worker is a single async loop: pull next, synthesize, play, signal `done` to that request's connection, repeat. While the worker is busy, incoming `speak` requests are pushed to the queue and the connection is sent `ack` immediately but `done` only after the worker reaches them. The single worker invariant guarantees no overlap.

Files: `src/core/daemon-server.ts` (new). Test: `daemon-server.test.ts` sends three rapid speak requests under `TELLME_TEST_SILENT=1` and asserts the `done` events arrive in order, never overlap.

## AC-5 (added): models stay loaded

Approach: Daemon constructs `new TellMeTts(config)` once at startup, calls `await tts.init()` once, and reuses that instance for every request. The instance is never `free()`'d while the daemon runs. Test asserts `init()` is called exactly once across N requests by spying.

Files: `src/core/daemon-server.ts` (new).

## AC-6 (added): stop halts and clears

Approach: Worker loop holds a reference to the active `audioHandle` (the streaming player handle from `audio-player.ts`). On `{kind: "stop"}`: clear the queue (rejecting each pending request's connection with `{kind: "stopped"}`), call `audioHandle.stop()`, mark the active request done (with `{kind: "stopped"}`), reply to the stopper with `{kind: "stopped"}`. The worker loop catches the abort and continues to the next iteration.

Files: `src/core/daemon-server.ts` (new). Test: enqueue a long utterance and two more, send `stop`, assert all three connections receive `stopped` and the daemon goes idle.

## AC-7 (added): request shape with optional overrides

Approach: `daemon-protocol.ts` defines the `SpeakRequest` schema. The daemon merges per-request fields onto its own defaults (sourced from `loadConfig()` at request time, not at startup — so config edits take effect on the next request without a daemon restart). Only `text` is mandatory; absent fields use the merged config. The merged config feeds the TTS engine and language detector exactly as the in-process CLI does today.

Files: `src/core/daemon-protocol.ts` (new), `src/core/daemon-server.ts` (new), `src/core/daemon-client.ts` (new — passes through CLI args). Test: send three requests with different `voice`/`speed`/`lang` overrides, assert each used the right values without polluting the others.

## AC-8 (added): synthesis errors surfaced

Approach: Worker wraps `tts.generateChunked(...)` in try/catch. On throw, send `{kind: "error", message: err.message}` on the request's connection, then close it. The CLI prints the message to stderr and exits 1. Test: pass an invalid `voice` and assert the client receives a useful error string before the connection closes.

Files: `src/core/daemon-server.ts` (new), `src/core/daemon-client.ts` (new).

## AC-9 (added): synchronous CLI

Approach: `daemon-client.ts:tryDaemonRoute` awaits the full `done` (or `error`/`stopped`) message before returning. The CLI's exit code reflects: `done.ok===true` → 0, `error` → 1, `stopped` → 130 (matches SIGINT convention). No early-return on `ack`.

Files: `src/core/daemon-client.ts` (new), `src/cli/index.ts` (modified).

## AC-10 (added): daemon sends completion signal

Approach: After `audioHandle.done` resolves for a request, the daemon writes `{kind: "done", ok: true}` to that request's connection, then closes write side. The client reads it and unblocks. Already covered structurally by the protocol design.

Files: `src/core/daemon-server.ts` (new).

## AC-11 (added): start | stop | status subcommands

Approach: In `src/cli/index.ts`, recognize `tellme daemon <subcmd>` as an alternate command path before the normal speak flow. Dispatch to `daemon-cmd.ts`:

- `start`: if socket alive → message "daemon already running"; else fork detached child running `node bin/tellme.js __daemon-main__` (a hidden flag the CLI handles by calling `runDaemon()` and never returning), wait until socket accepts a status ping, print socket path, exit.
- `stop`: connect, send `{kind: "status"}` to confirm it's our daemon, then close socket and signal SIGTERM to the PID; remove socket and pid files; print "daemon stopped".
- `status`: connect, send `{kind: "status"}`, print `running: true, socket: <path>, queue: <N>`. If no socket → print `running: false`.

Files: `src/cli/daemon-cmd.ts` (new), `src/cli/index.ts` (modified — subcommand dispatch + hidden `__daemon-main__` flag).
